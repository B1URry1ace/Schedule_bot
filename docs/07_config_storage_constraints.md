# 07. Конфигурация, хранение данных, ограничения и политика ошибок

## 1. Конфигурационные параметры

| Параметр | Значение по умолчанию | Назначение |
|---|---:|---|
| `timezone` | `Europe/Moscow` | единая таймзона для расписания |
| `schedule_days_default` | `7` | число дней для недельного просмотра |
| `schedule_cache_ttl_minutes` | `10` | срок жизни кэша расписания |
| `network_timeout_seconds` | `15` | таймаут HTTP-запроса к `.ics` |
| `voice_language` | `ru` | язык распознавания голосовых заметок |
| `temp_voice_dir` | `./` или `./tmp` | папка для временных voice-файлов |
| `max_note_length` | `4000` | безопасный лимит текста заметки |
| `telegram_message_soft_limit` | `3800` | запас до ограничения Telegram |

## 2. Что есть в текущей реализации

уже используются:
- `BOT_TOKEN`;
- `TIMEZONE = Europe/Moscow`;
- параметры локальной модели `LM_STUDIO_URL`, `LM_STUDIO_API_KEY`;
- имя модели Whisper `WHISPER_MODEL_NAME`;
- словарь соответствий групп и `attendeePersonId`.



## 3. Где храним локальные данные

### 3.1. SQLite база
Файл:
- `notes.db`

Таблицы:
- `users` — привязка пользователя к группе и `.ics`-ссылке;
- `notes` — заметки к событиям.

### 3.2. Временные voice-файлы
Файлы вида:
- `voice_<file_id>.ogg`

Жизненный цикл:
1. скачать из Telegram;
2. распознать;
3. удалить после обработки.


## 4. Политика кэша

### Что кешируем
- результаты `fetch_events(ics_url, start_date, days)`;


### Ключ кэша
Формат:
- `schedule:{hash(ics_url)}:{start_date}:{days}`

### TTL
- 10 минут по умолчанию.

### Когда кэш сбрасывается
- истёк TTL;
- пользователь явно обновил расписание;
- изменился источник `.ics`.

## 5. Ограничения и лимиты

### Платформа
- бот должен работать там, где доступен Python и aiogram;
- для локального распознавания голоса нужен CPU/GPU-ресурс и запущенный LM Studio;


### Сеть
- внешний `.ics`-источник может быть недоступен;
- Telegram file download может прерваться;
- локальный LM Studio API может не отвечать.

### Telegram
- длина сообщения ограничена;
- слишком длинные недельные сводки нужно разрезать на несколько сообщений.

### Данные
- одна заметка на одну пару для одного пользователя;
- некорректный `.ics` не должен ломать весь бот;
- неизвестная группа должна давать понятную ошибку.

## 6. Политика ошибок

### Что показываем пользователю
- короткий текст;
- понятное действие: «попробуйте ещё раз», «проверьте группу», «сервис недоступен»;
- без внутреннего стека и технических деталей.

### Что логируем
- тип ошибки;
- время;
- контекст: `user_id`, команда, `ics_url` или `event_uid`;
- техническую причину для диагностики.

### Что не логируем
- токены;
- содержимое секретов;
- персональные данные сверх необходимого минимума;
- лишние аудиофайлы после обработки.
